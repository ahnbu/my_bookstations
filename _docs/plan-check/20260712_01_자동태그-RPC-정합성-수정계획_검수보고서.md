### 1. 실행 불가 Placeholder 및 검증 불가 항목 (Critical)
계획 상에 명시적인 `TBD` 태그는 없으나, 에이전트가 실행할 수 없는 **"의미적 Placeholder(명령어 누락)"**가 다수 존재합니다. 서브에이전트는 브라우저 기반의 Supabase 대시보드 조작이 불가능합니다.

- ❌ **Task 3 Step 2 (검증 불가):** "Management API의 읽기 전용 SQL로 pg_get_functiondef를 조회해..."
  👉 실행할 `psql` 커맨드나 Node 쿼리 스크립트(dotenv 로드 포함)가 명시되지 않았습니다.
- ❌ **Task 3 Step 4 (검증 불가):** "Management API로 ... sql을 실행한다."
  👉 마찬가지로 에이전트가 어떻게 접속해서 SQL을 실행할지 방법이 없습니다.
- ❌ **Task 3 Step 5, 6 (검증 불가):** "읽기 전용 검증 SQL로 호출한다."
  👉 검증 쿼리문과 실행 환경이 누락되어 있습니다.

**💡 권장 조치:** Task 3의 모든 단계에 `@supabase/supabase-js`를 활용한 임시 Node 스크립트(스크래치 파일) 작성 지시 또는 로컬 `psql` 커맨드를 명시적으로 추가해야 합니다.

### 4. 크리티컬 리스크 및 직언
- **리스크 1 (실행 실패):** DB 연결 정보(Connection String, Service Key 등)를 `~/.env`나 `.env.local`에서 어떻게 읽어와서 SQL을 실행할 것인지 정의되지 않으면 Task 3 진입 시 즉시 에러가 발생합니다.
- **리스크 2 (기술 부채 잔존):** `구형 1인자 오버로드(get_books_by_tags(text[]))`를 제외 범위로 설정하고 무시했습니다. 당장 앱에서 안 쓰이더라도 DB 내에 중복/레거시 함수로 남아 있으면 추후 필터 관련 로직 수정 시 혼란을 야기할 수 있습니다. 운영 영향도가 낮다면 이번 기회에 `DROP FUNCTION` 처리를 고려하는 것이 좋습니다.

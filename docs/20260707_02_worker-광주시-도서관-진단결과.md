# Worker 광주시 도서관 진단결과

## 결론

- 운영 Worker는 같은 책 입력에서 광주 종이책과 시립소장 전자책을 15초 뒤 timeout으로 실패시킨다.
- 내 PC 직접 호출과 로컬 Worker 호출은 같은 광주시 endpoint를 2초 안팎 또는 1초 미만에 성공시켰다.
- 따라서 현재 확정 가능한 문제 정의는 “코드 parser나 15초 timeout 자체”가 아니라, “Cloudflare Worker 실행 위치에서 `lib.gjcity.go.kr:8443`, `lib.gjcity.go.kr:444`로 나가는 요청이 15초 안에 응답을 받지 못하는 문제”다.
- 나머지 3개 서비스(시립구독, e경기, e교육)는 이번 입력 기준으로 조용한 실패 증거가 없다. 다만 e경기 코드는 소장형/구독형 중 한쪽만 실패하면 전체 에러로 드러나지 않을 수 있는 구조적 위험이 있다.
- 30초/60초면 성공하는지 여부는 아직 확정하지 못했다. 다만 이 값은 해결책 후보가 아니라 “차단/무응답인지, 단순 지연인지”를 구분하기 위한 관측값으로만 취급한다. 15초를 넘기는 방식은 사용성 기준상 대응안에서 제외한다.

## 나머지 서비스 점검

운영 Worker 응답과 원본 endpoint 직접 호출을 함께 확인했다.

| 서비스 | 운영 Worker | 직접 호출 | 판정 |
|---|---|---|---|
| 시립구독 전자책 | `1 / 1` | HTTP 200, 53,384 bytes, 결과 1건 | 조용한 실패 증거 없음 |
| e경기 소장형 | `0 / 0` | HTTP 200, `httpStatus: OK`, 결과 0건 | 조용한 실패 증거 없음 |
| e경기 구독형 | `0 / 0`에 포함 | HTTP 200, 결과 목록 `null`, 총 0건 | 조용한 실패 증거 없음 |
| e교육 성남 | `errorCount: 0` | HTTP 200, “찾으시는 자료가 없습니다”, 결과 0건 | 조용한 실패 증거 없음 |
| e교육 통합 | `errorCount: 0` | HTTP 200, “찾으시는 자료가 없습니다”, 결과 0건 | 조용한 실패 증거 없음 |

주의할 점은 e경기 통합 함수가 소장형과 구독형을 `Promise.allSettled`로 실행한 뒤, 둘 다 실패할 때만 전체 실패를 던진다는 점이다. 한쪽만 실패하면 0건 또는 부분 결과처럼 보일 수 있다. 이번 입력에서는 두 하위 endpoint 모두 정상 0건이라 실제 조용한 실패는 관측되지 않았다.

## 원격 Cloudflare 계측 시도

| 단계 | 결과 | 의미 |
|---|---|---|
| `wrangler whoami` | API token 환경변수 주입 후 성공 | 인증 자체는 가능 |
| remote dev 전체 측정 | 7분 이상 응답 미완료, 결과 JSON 없음 | 한 요청에서 15/30/60초 6개 측정을 묶는 방식은 remote dev에 부적합 |
| remote dev 분할 측정 | `/workers/subdomain/edge-preview` Cloudflare API 요청 실패 | 현재 토큰 권한 또는 Cloudflare remote preview API 단계가 막힘 |
| 임시 Worker 배포 전 security gate | `.env.example` 민감 파일 포함으로 차단 | 배포는 중단함 |

이 때문에 “Cloudflare에서 30초 또는 60초 안에는 응답이 오는가”는 이번 실행에서 확정하지 못했다. 단, 응답이 오더라도 timeout 연장은 해결책으로 채택하지 않는다.

## 최종 정리 상태

- 진단 route와 임시 source test는 작업트리에서 제거했다.
- `library-checker/src/index.ts`에는 최종 diff가 남지 않았다.
- `npx tsc --noEmit`은 실패했다. 오류는 `@supabase/auth-js`, `@supabase/realtime-js`, `@supabase/storage-js` 타입이 DOM/Node 타입(`BroadcastChannel`, `Storage`, `Buffer`, `NodeJS` 등)을 찾지 못하는 기존 환경 이슈다.
- 로컬/원격 진단 원본은 `library-checker/temp/` 아래에만 남겼다.

## 코드와 변경 이력에서 확인한 점

- `DEFAULT_TIMEOUT`은 15초이며 광주 종이책, 시립소장, e경기, e교육 fetch에 공통으로 적용된다.
- 광주 종이책 함수는 `https://lib.gjcity.go.kr:8443/kolaseek/plus/search/plusSearchResultList.do`로 ISBN 상세 POST를 보낸다.
- 시립소장 함수는 `https://lib.gjcity.go.kr:444/elibrary-front/search/searchList.ink`로 검색어 GET을 보낸다.
- 운영 요청은 광주 종이책, e교육, e경기, 시립전자책을 거의 동시에 시작한다. 광주 종이책과 시립소장이 각각 15초 timeout으로 실패하기 때문에 전체 응답도 약 16초에 끝난다.
- `hasCacheBlockingError`가 에러 포함 응답을 캐시하지 않도록 막고, 실제 운영 응답도 `Cache-Control: no-store`였다. 캐시된 실패가 반복 표시되는 문제는 이번 현상과 맞지 않는다.
- 2026-05-03 백로그에 이미 “광주 종이책 Worker 타임아웃 원인 분리”가 남아 있다. 시립구독 전자책은 같은 날 불필요한 세션 쿠키 제거로 복구된 별도 이슈다.

## 문제 정의

현재 문제는 “광주시 계열 사이트가 느리다”로 정의하면 부정확하다. 내 PC와 로컬 Worker에서는 빠르게 성공하기 때문이다.

더 정확한 정의:

> Cloudflare Worker 운영 실행 위치에서 광주시립도서관 종이책 KOLAS(`:8443`)와 시립소장 전자책(`:444`) 요청이 15초 안에 응답을 받지 못한다. 같은 요청은 로컬 네트워크와 로컬 Worker에서는 성공하므로, parser drift나 로컬 요청 구성 문제보다는 Cloudflare 실행 위치와 대상 서버 사이의 연결/차단/응답 지연 문제로 분리해야 한다. 15초 초과 응답은 제품 사용성 기준상 실패로 본다.

## 대응 방향

1. Cloudflare 원격 계측을 먼저 살린다.
   - remote preview 권한 문제를 해결하거나, security gate의 `.env.example` false positive를 판단한 뒤 별도 이름의 token 보호 임시 Worker로 측정한다.
   - 30초/60초 계측은 해결책 검증이 아니라 원인 분리다. 응답이 오면 “단순 지연”, 끝까지 안 오면 “차단/무응답”으로 분류한다.

2. 광주시 계열은 15초 안에 끝나는 별도 처리 전략으로 분리한다.
   - 후보: Vercel/Node API, Supabase Edge Function, 별도 경량 proxy, 또는 로컬/서버 기반 scheduled fetch 후 캐시 저장.
   - Cloudflare Worker는 결과 취합과 캐시 응답만 담당하게 분리하는 편이 낫다.

3. 30초/60초에서 성공하더라도 timeout 연장은 하지 않는다.
   - 전체 API를 오래 붙잡지 말고, 광주시 계열은 “갱신 중/나중에 갱신” 흐름이나 사전 갱신 캐시로 빼는 것이 사용자 경험상 낫다.

4. e경기 부분 실패는 별도 개선 후보로 남긴다.
   - 소장형/구독형 중 한쪽만 실패할 때 `errors` 필드를 남기도록 하면 0건과 부분 실패를 구분할 수 있다.

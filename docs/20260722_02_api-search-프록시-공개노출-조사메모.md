# `/api/search` 프록시 공개 노출 — 조사 메모

> 조사만 수행한 메모다. **코드 변경 없음.** 결론은 "지금은 조치하지 않는다"이다.

## 1. 요약

| 항목 | 내용 |
|---|---|
| 발단 | TTB 키가 클라이언트 번들에 노출됐다는 의심 |
| 조사 결과 | ✅ **키 노출은 오탐**. 배포 번들에 키 없음 |
| 부수 발견 | ⚠️ `/api/search`가 무인증·무제한 공개 상태 |
| 최종 판단 | **현행 유지.** 관찰만 하고 코드는 건드리지 않는다 |
| 판단 근거 | 사용자 10인 이하 / 유료 API 아님 / 최악이 일시 중단 |

## 2. 오탐이었던 건 — TTB 키 노출

### 의심 지점

`services/aladin.service.ts:3`에 다음 코드가 있다.

```ts
const TTB_KEY = import.meta.env.VITE_ALADIN_TTB_KEY;
```

Vite의 `VITE_` prefix 변수는 통상 빌드 시 번들에 평문 삽입되므로, 이것만 보면 노출로 보인다.

### 실제로는 노출되지 않았다

배포된 번들을 직접 받아 검사했다.

```bash
curl -s https://my-bookstations.vercel.app/ | grep -oE '/assets/[^"]+\.js'
# → /assets/index-BqyriyD2.js

curl -s "https://my-bookstations.vercel.app/assets/index-BqyriyD2.js" -o deployed.js
KEY=$(grep "^VITE_ALADIN_TTB_KEY=" .env.local | cut -d= -f2)
grep -qF "$KEY" deployed.js && echo "노출됨" || echo "미포함"
# → 미포함
```

| 검사 항목 | 결과 |
|---|---|
| 번들 크기 | 608,234 bytes |
| `ttbkey` 문자열 | ✅ 0건 |
| **실제 키 값(22자)** | ✅ **미포함** |
| `/api/search` 경로 | ✅ 1건 |

### 왜 안 들어갔나

같은 파일에 이미 환경 분기가 있었다.

```ts
const IS_PRODUCTION = process.env.NODE_ENV === 'production';
const ALADIN_API_URL = IS_PRODUCTION ? '/api/search' : '/ttb/api/ItemSearch.aspx';

if (!IS_PRODUCTION) {
  params.append('ttbkey', TTB_KEY);   // 로컬에서만 키 사용
}
```

빌드 시 `IS_PRODUCTION`이 상수 `true`로 치환되고, minifier가 `if (!true) {...}`를 dead code elimination으로 제거한다. `TTB_KEY`를 참조하는 유일한 코드가 사라지면서 변수 자체가 트리셰이킹으로 빠졌다.

배포 번들의 실제 컴파일 결과는 삼항 분기조차 남지 않았다.

```js
const Kw="/api/search", Sr=async(r,s,n=1,i=40)=>{ ... }
```

**설계 의도대로 정확히 동작한 것이다.** 두 변수는 이름이 유사할 뿐 역할이 다르다.

| 변수 | 위치 | 용도 |
|---|---|---|
| `VITE_ALADIN_TTB_KEY` | `.env.local` | **로컬 개발 전용.** Vite 프록시로 알라딘 직접 호출 |
| `ALADIN_TTB_KEY` | Vercel 환경변수 | **프로덕션 전용.** `api/search.ts`가 서버에서 첨부 |

## 3. 실제로 남는 건 — 프록시 공개 노출

키는 안전하나, **키의 사용 권한이 공개**돼 있다.

```bash
curl -s "https://my-bookstations.vercel.app/api/search?Query=test&QueryType=Keyword&MaxResults=1" -w "status=%{http_code}"
# → status=200
```

- `api/search.ts`에 인증·rate limit·Referer 검증이 없다.
- CORS 헤더가 없어 브라우저 타 도메인 fetch는 막히지만, `curl`·서버 스크립트에는 무력하다.
- 부차적으로 알라딘 호출이 `http://` 평문이다 (`api/search.ts:4`).

## 4. 그래서 왜 조치하지 않나

### 리스크 실측

| 항목 | 값 |
|---|---|
| 사용자 규모 | 본인 포함 **10인 이하** |
| API 과금 | ❌ 없음 (무료) |
| 알라딘 일일 한도 | **5,000회/일** (일반 API) |
| 한도 초과 시 | 당일 조회 실패 → 익일 자동 복구 |
| 최악 시나리오 | ⚠️ **일시적 검색 중단**. 금전 손실·데이터 유출 없음 |
| 현재 남용 정황 | 관측된 바 없음 |

핵심은 **유료 API가 아니라는 점**이다. 남용당해도 청구서가 날아오지 않는다. 한도를 소진당하면 그날 검색이 안 되고, 다음 날 자동으로 풀린다.

### 검토했다가 기각한 안 — Referer 검증

처음엔 "10줄이면 되는 쉬운 조치"로 판단했으나, plan-check-lite 검수에서 **수정 필요** 판정이 나왔다.

| 문제 | 내용 |
|---|---|
| `Origin` 헤더 미전송 | same-origin **GET**에 브라우저는 `Origin`을 붙이지 않는다. 이 앱 호출은 전부 same-origin GET이라 항상 빈 값 |
| Vercel 프리뷰 차단 | 배포마다 생성되는 `my-bookstations-<hash>.vercel.app`을 하드코딩 허용목록이 전부 막는다 |
| 직접 접속 차단 | 주소창 입력·북마크는 Referer가 없어 403. 디버깅 시 본인도 막힘 |
| 방어 효과 낮음 | `curl -H "Referer: ..."` 한 줄로 우회 가능 |
| 롤백 계획 부재 | 오탐 시 **live 검색 기능 전체 중단** |

정리하면 **얻는 것(위조 가능한 약한 방어) 대비 잃는 것(live 서비스 중단 리스크)이 크다.** 문제가 있는지도 모르는 상태에서 동작하는 서비스에 차단 로직을 넣는 건 순서가 뒤바뀐 판단이다.

## 5. 결론과 트리거

**현행 유지.** 단, 아래 조건이 관측되면 재검토한다.

| 트리거 | 조치 |
|---|---|
| 알라딘 조회가 이유 없이 실패하기 시작 | 한도 소진 여부 확인 → 남용이면 rate limit 도입 |
| 일일 사용량이 평소 대비 비정상 증가 | 동일 |
| 서비스를 외부에 공개 홍보하게 됨 | 사전에 rate limit 검토 |

재조치 시에는 **Referer 검증을 건너뛰고 rate limit(Upstash 무료티어)으로 직행**한다. Referer는 중간 단계로서 가치가 낮다는 것이 이번 검수의 결론이다.

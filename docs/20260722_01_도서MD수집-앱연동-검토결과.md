# 도서 MD 수집 — 앱 연동 검토 결과

- 작성일: 2026-07-22 (KST)
- 대상: `book-info-collector` 스킬과 My Bookstation 연동
- 상태: 검토 완료 / **구현 없음 권장**

## Context

`book-info-collector` 스킬을 앱의 책에 버튼으로 붙여, 클릭 시 해당 책의 MD를 다운로드받게 할 수 있는지 검토 요청.

실사용 목적:
- **강의 리서치용 도서 자료 수집**
- 필요 범위: **목차 / 저자소개 / 책소개 전문**
- 사용 기기: **주로 PC**

## 결론

| 안 | 판정 | 근거 |
| :--- | :--- | :--- |
| 원안 (앱 버튼 → MD 다운로드) | ❌ 권장 안 함 | 서버 스크래퍼 포팅 필요, 정본 이원화 |
| 1차 대안 (앱에 "수집 URL 복사" 버튼) | ❌ 철회 | 기존 링크복사로 이미 가능, 서재 밖 책 커버 못 함 |
| **최종 (CLI `search` → `collect` 그대로)** | ✅ | **추가 구현 0. 이미 동작함** |

## 검토 1: 원안은 기술적으로 가능하나 비용이 큼

앱의 도서 데이터는 알라딘 **ItemSearch API** 결과(`types.ts:12` `AladdinBookItemSchema`)로,
필드는 `title / author / pubDate / description / isbn13 / cover / price / publisher / link`뿐이다.
**목차·저자소개·책소개 전문은 API 응답에 애초에 없다.**

스킬이 그 3개 섹션을 얻는 방식은 API가 아니라 **HTML 스크래핑 4회**다
(`skills/book-info-collector/scripts/lib/aladin-page.mjs:299` `fetchBookSource`):

1. 상세 페이지 HTML → `contentKey` 추출 (`extractContentKey`, 정규식 9종 폴백)
2. `getContents.aspx?ISBN={key}&name=Introduce` (목차)
3. `...&name=AuthorInfo` (저자소개)
4. `...&name=PublisherDesc` (책소개)

2~4는 1단계의 **`set-cookie`를 릴레이하고 `referer`를 세팅해야** 응답한다 (`aladin-page.mjs:303-312`).

→ 브라우저에서 직접 불가. **CORS 차단 + 쿠키 릴레이 불가** → Worker/Vercel 경유 필수.
즉 "버튼 추가"가 아니라 **스크래퍼 백엔드 포팅**이다.

가장 큰 비용은 개발 공수가 아니라 **정본 이원화**다.
알라딘 HTML이 바뀌면 CLI와 Worker 두 벌을 각각 고쳐야 한다.
`contentKey` 추출이 9단 폴백인 것 자체가 알라딘 마크업 불안정성의 증거다.

## 검토 2: "수집 URL 복사" 버튼도 불필요 (1차 대안 철회)

사용자 지적으로 철회. 실제 검증한 근거:

1. **개별 복사는 이미 됨** — 모달 상세 / "종이책" 우측 메뉴에서 링크 복사 가능
2. **서재 밖 책을 커버 못 함** — 수집 대상이 항상 내 서재에 있지 않음. 알라딘에만 있는 책도 있음.
   서재에 먼저 저장해야만 수집 가능한 구조는 불필요한 선행 단계를 강제한다.
3. **`user_library`는 RLS 보호 + 사용자별 스코프** (`stores/useBookStore.ts:378` `.eq('user_id', session.user.id)`).
   anon key로 조회 시 0행 반환 확인 → 앱 밖에서 일괄 처리하려면 어차피 세션이 필요하다.

## 검토 3: 실측 — CLI가 이미 요청한 워크플로우를 커버함

사용자가 제안한 **"이런 책 있는지 조사 → 리스팅 → 이거 저장"** 흐름은
`search` → `collect`로 **이미 완결된다.** 추가 구현이 필요 없다.

### `search`가 "수집 URL" 컬럼을 이미 출력한다

```
node .../book-info-collector.mjs search "강의" --max-results 6 --env D:/vibe-coding/my_bookstation/.env.local
```

출력 컬럼: `번호 | 도서명 | 저자 | 출판사 | 출간일 | ISBN13 | 짧은 소개 | 수집 URL`

→ 리스팅 결과에서 원하는 번호만 골라 URL을 `collect`에 넘기면 끝.

### `search` URL이 `collect`에 그대로 들어간다 (검증됨)

`search` 출력 URL에는 `&amp;partner=openAPI&start=api`가 붙어 있는데,
이를 **가공 없이 그대로** 넘겨도 정상 수집된다.

```
node .../book-info-collector.mjs collect \
  "https://...ItemId=365659530&partner=openAPI&start=api" \
  "https://...ItemId=337406511" \
  "https://...ItemId=354080436" \
  --separate --dry-run
```

결과: 3권 모두 `목차/저자소개/책소개: full`, `검수: pass`

### 실측 성능 (34권 표본, 8개 장르)

| 항목 | 실측값 |
| :--- | :--- |
| **34권 배치 소요** | **84~114초** (2회 측정) |
| **권당 평균** | **약 2.5~3.4초** |
| 권당 편차 | 0.8s ~ 24s (알라딘 응답 지연 영향) |
| MD 1건 크기 | 4~17KB |
| 전권 검수 | **34/34 pass** |

> 동일 표본 2회 실행에서 84초 / 114초로 **36% 편차**가 났다.
> 알라딘 응답 속도에 좌우되므로 **권당 3.5초 기준으로 여유 있게 계획**할 것.

표본은 `클로드 코드 / 한국사 / 요리 / 물리학 / 시집 / 경제경영 / 그림책 / 번역서 소설`
8개 장르에서 알라딘 ItemSearch로 수집해 장르 편향을 줄였다.

> 100권이면 약 6분, 500권이면 약 29분. (권당 3.5초 기준)

### ⚠️ 중요: CLI 시간 ≠ 사용자 체감 시간

**스킬 호출로 수집하면 CLI 실행 시간과 무관하게 훨씬 오래 걸린다.**

실제 세션 기록 (`codex:019f54b3-1dc7-7d72-8587-496dabb6bfd6`, 2026-07-12):

| 측정 대상 | 값 |
| :--- | :--- |
| CLI가 보고한 시간 | 1.3~1.5초 |
| **실제 세션 소요** | **56분 / 3권** |
| **토큰 소모** | **267.7K** |

원인은 fetch가 아니라 **AI 왕복**이다. 스킬 호출 1건당:
1. SKILL.md + 프로젝트 규칙 확인 (`exec`)
2. CLI 실행 (`exec`) ← 여기만 2.5~3.4초
3. 검증·reviewer 결과 해석 후 보고

권당 약 90K 토큰이 전부 이 왕복 비용이다.
**CLI가 출력하는 `n초`는 fetch+파싱 시간일 뿐이며, 사용자 체감 시간을 대표하지 않는다.**

### 품질: 스킬 경유 vs CLI 직접 — 차이 없음 (검증됨)

### 🐛 발견·수정한 버그 — 짧은 목차 누락 (2026-07-22)

**최초 측정에서 목차 `missing`이 12권이었으나, 그중 5권은 실제로 목차가 존재했다.**

| 목차 길이 | 존재하는 id |
| :--- | :--- |
| 긴 목차 | `div_TOC_All` + `div_TOC_Short` |
| **짧은 목차** | **`div_TOC_Short`만** |

## 검토 4: "대량 선수집 후 로컬 조회" 접근의 가치

사용자 제안: 책 정보를 미리 대량으로 MD 저장 → 리서치·강의 때 바로 조회, 신규분만 증분 추가.

### 유리한 점 (실측 근거)

- **용량 부담 없음** — 권당 평균 약 12KB. 500권이라도 6MB 수준. 로컬 저장에 전혀 부담되지 않는다.
- **증분 추가가 이미 설계에 있음** — 스킬이 ISBN 동일성으로 충돌을 판정하고
  완전한 MD/source 쌍만 교체한다 (`SKILL.md` "staged pair replacement with rollback").
  같은 책 재수집 시 중복이 쌓이지 않는다.
- **조회는 이미 됨** — 출력 폴더가 `C:/Users/ahnbu/cowork/download/_book_info_collect`로,
  **cowork는 옵시디언 볼트**다. 즉 대량 수집만 해두면 별도 뷰어 없이 옵시디언에서 바로 검색·링크된다.

### 불리한 점 / 유보 사항

- **소비처가 아직 불명확** — 사용자도 인정한 부분. 선수집은 "쓸지 모르는 것에 미리 비용을 내는" 구조다.
- **선수집 비용이 작지 않음** — 권당 약 3.5초. 서재 전권을 밀어 넣으면 수십 분간 알라딘에 연속 요청이 나간다.
- **신선도** — 가격·재고성 정보는 시간이 지나면 어긋난다.
  다만 이번 용도(목차·저자소개·책소개)는 **거의 불변**이라 실질 문제는 작다.

### 권장 — 대량 선수집에 **찬성**으로 수정

> 초기 검토에서는 "필요할 때 뽑는 게 싸니 전량 선수집 반대"였다.
> 그 근거가 **권당 5.8초 + 스킬 호출 비용 미반영**이라는 잘못된 측정이었으므로 **철회한다.**

실측이 바꾼 판단:

| 방식 | 34권 기준 | AI 왕복 |
| :--- | :--- | :--- |
| 스킬로 한 권씩 | **수십 분 + 약 3M 토큰** | 34회 |
| **CLI 배치 1회** | **84~114초 + 왕복 1회** | **1회** |

비싼 것은 수집이 아니라 **AI 왕복**이다.
따라서 **한 권씩 스킬로 부르는 것이 최악이고, 몰아서 CLI로 돌리는 것이 최선**이다.
이는 사용자가 제안한 **"대량 선수집 후 로컬 조회"에 유리한 근거**다.

- 500권 선수집 = 약 29분 / 약 6MB → **1회 배치로 끝난다**
- 이후 조회는 옵시디언에서 즉시 (토큰 0)
- 신규분만 추가할 때도 URL 여러 개를 한 번에 넘기면 왕복 1회

## 최종 권장 워크플로우 (추가 구현 0)

**핵심: 스킬로 한 권씩 부르지 말고, 터미널에서 CLI로 몰아서 실행할 것.**

```bash
# 1) 후보 리스팅 (수집 URL 컬럼이 함께 나옴)
node C:/Users/ahnbu/.claude/skills/book-info-collector/scripts/book-info-collector.mjs search "<주제>" --env D:/vibe-coding/my_bookstation/.env.local

# 2) 고른 URL을 한 번에 배치 수집 (여러 개를 공백으로 나열)
node C:/Users/ahnbu/.claude/skills/book-info-collector/scripts/book-info-collector.mjs collect <URL1> <URL2> <URL3> ... --separate

# 3) 옵시디언 cowork 볼트에서 바로 조회·인용
```

URL이 많으면 파일로 두고 한 번에 넘긴다:

```bash
node C:/Users/ahnbu/.claude/skills/book-info-collector/scripts/book-info-collector.mjs collect $(cat urls.txt | tr '\n' ' ') --separate
```

앱 변경 없음. Worker 변경 없음. 정본은 CLI 한 벌.
품질은 스킬 경유와 **바이트 단위로 동일**함이 검증되었다.

## 원안을 재검토할 조건

아래가 참이 되면 앱 연동을 다시 검토할 가치가 있다.

- 모바일에서 수집해야 할 실제 필요가 생김
- 수집 MD를 앱 안에서 보관·검색해야 함 (다운로드가 아니라 저장 기능)
- 본인 외 사용자에게 이 기능을 제공해야 함

현재 셋 다 해당하지 않는다.

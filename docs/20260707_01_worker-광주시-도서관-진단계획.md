# Worker 광주시 도서관 진단작업 계획

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Cloudflare Worker 실행 환경에서 광주시 도서관 계열 조회가 15초 안에 실패하는 원인을 endpoint, 실행 위치, 요청 구성 단위로 분리한다. 30초/60초 계측은 원인 관측용이며, 제품 대응안에서 timeout 연장은 제외한다.

**Architecture:** 운영 조회 로직은 변경하지 않고, 임시 진단 route 또는 임시 Worker에서 광주 종이책 KOLAS와 시립소장 전자책 endpoint만 계측한다. 진단 route는 DB를 쓰지 않고 JSON만 반환하며, 모든 응답은 `Cache-Control: no-store`로 처리한다. 진단이 끝나면 route 제거 또는 임시 Worker 삭제를 완료 조건에 포함한다.

**Tech Stack:** Cloudflare Workers, TypeScript, Wrangler, PowerShell, Node.js, `node-html-parser`

## 현재 관측 사실

- 내 PC에서 원본 광주 종이책 KOLAS POST는 2초 안팎으로 HTTP 200과 검색결과 HTML을 반환했다.
- 내 PC에서 원본 시립소장 전자책 GET은 1초 미만으로 HTTP 200과 0건 처리 가능한 HTML을 반환했다.
- 프로덕션 Worker POST는 같은 책 입력으로 3회 연속 약 15.5초 뒤 `gwangjuPaper` timeout과 `siripEbook.errors.owned` timeout을 반환했다.
- 시립구독 전자책은 같은 Worker 호출에서 `1/1`로 성공했다.
- e경기 전자책은 같은 Worker 호출에서 `0/0` 정상으로 보였다.
- e교육은 같은 Worker 호출에서 대체로 `errorCount: 0`이지만, 반복 호출 중 1회 `errorCount: 1`이 관측됐다.
- `wrangler whoami --env-file C:/Users/ahnbu/.env`는 현재 `Not logged in`으로 실패했다. Cloudflare 원격 로그와 remote dev 검증은 인증 상태 복구가 선행 조건이다.

## 진단 질문

1. Cloudflare Worker에서 `lib.gjcity.go.kr:8443`과 `lib.gjcity.go.kr:444`가 15초 안에 왜 응답하지 못하는가?
2. 실패가 HTTP status 기반 실패인가, 네트워크 연결/응답 대기 timeout인가?
3. HTTP 200이 오더라도 body가 정상 HTML인가, 방화벽/차단/에러 페이지인가?
4. body는 정상인데 parser selector가 결과를 놓치는가?
5. 운영 Worker의 통합 조회 구조가 하위 endpoint 실패를 지나치게 오래 기다리거나 잘못 표시하는가?
6. 광주시 계열을 15초 안에 안정적으로 처리하려면 Worker 밖의 실행 위치로 분리해야 하는가?
